# Upgrade: projects v1.x → v3.0.0

## Summary

v3.0.0 is a major release that migrates all shared project data operations to the remote filesystem model and adds 8 new API members. This is a combined migration — there was no v2.0.0 release for this collection. Requires agent-index-core v3.0.0.

---

## Breaking Changes

1. **Remote filesystem for shared data**: All shared project data operations (`/shared/projects/`) now use `aifs_*` remote filesystem tools instead of direct file access.
2. **`members-registry.json` access changed**: Member lookups now use `aifs_read("/members-registry.json")` instead of local file reads.
3. **`collection-setup-responses.md` access changed**: Setup responses on the remote filesystem are now read via `aifs_read`.
4. **Setup parameter descriptions updated**: "filesystem path" references changed to "remote filesystem path".

---

## New API Members in v3.0.0

The following capabilities were added and will be offered to members during the update flow:

- `create-idea` — create private ideas in member workspace for later sharing
- `share-idea` — promote private ideas to project shared space
- `manage-ideas` — view, edit, and manage ideas with artifact promotion
- `update-project` — incremental project updates with activity log
- `manage-action-items` — track action items with dependencies and status workflows
- `project-decide` — record project decisions with rationale
- `channel-digest` — summarize project channel conversations
- `project-pulse` — status reports and conversational project Q&A

---

## Migration Steps

### Step 1 — Prerequisite

Ensure agent-index-core has been upgraded to v3.0.0 first. The projects v3.0.0 depends on the remote filesystem infrastructure.

### Step 2 — Admin: Migrate Project Data

1. Upload existing project files from the old shared mount to the remote filesystem at `/shared/projects/` using `aifs_write`
2. Verify project files are accessible: `aifs_list("/shared/projects/")` should list all existing projects
3. For each project, verify `project.md` and associated files are readable via `aifs_read`

### Step 3 — Admin: Publish Update Instructions

Run `@ai:publish-updates` after upgrading the projects collection on the remote filesystem. The generated `collection-update` entry will include `api_changes` listing the 8 new members.

### Step 4 — Members: Apply Update

Members run `@ai:apply-updates`, which will:

1. Update existing task definitions (`create-project`, `edit-project`, `archive-project`, `unarchive-project`) to v3.0.0
2. Offer the 8 new capabilities for installation
3. Re-run setup for tasks whose parameters changed (path references)

---

## Preserved Member Data

- All existing project data is preserved (it moves from local shared mount to remote filesystem during admin migration)
- Member setup responses for `create-project`, `edit-project`, `archive-project`, `unarchive-project` are preserved, but path parameters may need updating from local paths to remote paths
- `projects-tutorial` skill state is preserved

---

## Parameters Requiring Member Attention

- `shared_projects_path` — if members had custom overrides pointing to local filesystem paths, these need to be updated to remote filesystem paths. The default value is updated automatically.
